## Executive verdict

**omx follows OMC's analysis-lane structure with high fidelity where it chose to follow it, and is at parity-or-better on every mechanism it re-implemented; the deficit is confined to operational robustness of the autonomous loop, and the earlier single-pass gap analysis — because it validated whole subsystems rather than drilling into sub-mechanisms — missed several concrete, in-scope defects that this deeper pass surfaces.** Where omx elected to mirror OMC, it did so faithfully: the evaluator JSON contract reproduces all four loud-fail throws with the same message strings (`evaluator.py:27-50` vs `contracts.ts:178-201`); the keep-policy decision tree reproduces OMC's ordering including the bootstrap-keep branch and the commit-140fd064 fix (`decision.py:50-91`); the 3-artifact ledger writes a byte-identical TSV header (`ledger.py:25`); the append-merge wiki reproduces OMC's confidence-max semantics with the same `>=` rule (`ingest.py:66-82`); and the path module validates every id with anchored regexes and atomic-renames every write (`omx_paths.py`).

**omx is genuinely better than OMC in a cluster of places, and these are worth the maintainer knowing they can defend.** On knowledge removal, omx is stricter — no direct delete verb at all, removal only through a git-guarded two-phase gc that validates the whole plan before mutating (`gc.py:191-241`), against OMC's raw `wiki_delete`. It adds a Jaccard near-duplicate lint check OMC lacks (`lint.py:102-128`), a fresh-page orphan exemption OMC lacks (`lint.py:36-46`), and query/lint/index paths that tolerate a corrupt page instead of crashing. Its CLI is JSON-always with non-finite floats normalized to `null` (`cli.py`, `allow_nan=False`) — a stronger machine-consumability contract than OMC's mixed human/artifact output. And on the anti-self-approval question that ralph solves with a correlated reviewer token, omx's out-of-process deterministic `evaluator.sh` is a stronger objective truth-source than another LLM's provenance-checked prose, so importing a reviewer-subagent gate would be both invariant-violating and weaker — correctly out of scope.

**The shallow prior analysis caused three classes of miss.** First, it folded distinct half-mechanisms into single coarse rows: "HAS evaluator contract" swallowed the eval-error-as-discard signaling and the CLI's rc-1-on-error half-mechanism; "Plugin install/manifest/HUD" was one blanket NA row that hid a live version-drift defect (`plugin.json`=0.1.14 vs `pyproject.toml`=0.1.0, never bumped across 14 releases, README hardcodes 0.1.0 in three spots). Second, it treated whole subsystems as absent when omx has them in a different shape: it marked capture-then-curate ABSENT when omx has it split by role-in-one-pass rather than by time, and it implied `--from-report` auto-ingests when the code shows it prints candidates and writes nothing. Third, and most consequential, it validated NA dismissals at the subsystem level and so never saw the sub-mechanism gaps — including the single most severe finding in the audit, which the prior pass never surfaced at all: **there is no sealed-evaluator guard**, so the loop trusts a score from an `evaluator.sh` that its own author can edit mid-loop with nothing detecting it (no `sha256`/`sealed`/`integrity` code exists in core). The same subsystem-level blind spot hid an unbounded TensorBoard ingest that directly contradicts the repo's own stride-extraction rule, the total absence of author-vs-reviewer separation on the gated `report.md` deliverable, an `omx clean` verb a shipped skill references but that does not exist, and the absence of any loop-completion visibility marker for the leaving-work autonomous case. Meanwhile the four biggest prior-flagged holes (concurrency lock, git-revert primitive, confidence-gated handoff, capture write path) are all re-confirmed still-absent, with the concurrency lock sharpened: `state.py`'s `load_state`/`save_state` are wired into no production or skill code (they carry test-only references) — the state envelope is vestigial.

## New findings

This is the headline section: everything the prior gap file did not carry (or carried wrongly) that survived adversarial verification. Findings are grouped by cluster. Corrections to the prior gap file carry an explicit "prior said X, actually Y" line. Two verdicts arrived CORRECTED and are used as corrected: knowledge's structural-contradiction (grouping-axis nuance folded in) and knowledge's capture-then-curate (status corrected HAS→PARTIAL).

### loop-contracts

**Candidate-artifact worker-exit protocol / staleness validation — NA_QUESTIONABLE, verified CONFIRMED.**
OMC's supervisor validates a worker's `candidate.json {status, base_commit, candidate_commit}`, asserting `candidate_commit` resolves == HEAD and `base_commit == last_kept_commit` before trusting the candidate (`09-research.md:121,183` / `runtime.ts:1157-1277,818-819`). omx has no worker session — its "candidate" is a checkpoint a human already trained, graded in-process (`skills/exp-loop/SKILL.md:74-88`) — so the exit-protocol artifact is a defensible v0.1 scope omission. But the staleness-validation idea has a real omx shape: nothing checks that the graded checkpoint was produced from `last_kept_commit`, so a human can grade a checkpoint from a stale/wrong config and the ledger silently records it as an improvement over an unrelated baseline. Adoption: `omx eval --expect-base-commit <last_kept>` that git-rev-parses the checkpoint's recorded training commit (from a launch-written sidecar) and loud-fails on mismatch, mirroring `runtime.ts:1246-1277`. This is the confounding-guard the repo's own minimum-change-revert rule wants.

**Iteration-error failure classification — ABSENT, verified CONFIRMED.**
OMC distinguishes a run-FATAL iteration error (`failAutoresearchIteration` → run status `failed`, operator must intervene) from TRANSIENT outcomes (`noop` relaunches; `interrupted` runs a reset-safety probe) at `runtime.ts:670-695,1279-1346`. omx collapses every evaluator fault — non-zero exit, timeout, empty stdout, unparseable line — into one `status='error'` that the decision tree turns into discard (`evaluator.py:82-116`, `decision.py:69-71`). A flapping or crash-looping evaluator therefore discards forever, indistinguishable from a legitimate fail, with no fatal-stop. Adoption: add a `fault_class` field ('timeout'|'nonzero_exit'|'unparseable'|'empty') to the error record and a loop-level circuit — a new loop-health verb that reads the ledger tail and loud-fails after N consecutive `status='error'` iterations ("evaluator appears broken — fix evaluator.sh before continuing"). Keeps no-hooks: it is a CLI verb `exp-loop` step-3 calls.

**eval-error-as-discard signaling / rc-1 half-mechanism — HAS, verified CONFIRMED.**
Prior said: this was folded into one "HAS evaluator contract" row. Actually: it is a distinct half-mechanism worth naming. omx's runner is fault-tolerant-by-recording — it captures the fault into the `EvaluationRecord` (never raises) so `decide_outcome` turns it into discard (`evaluator.py:97-116`, `decision.py:69`). It is better than OMC in one respect: the CLI surfaces `rc=1` specifically when the evaluator itself broke vs `rc=0` when it produced a graded verdict (`cli.py:143-162`, test `test_eval_evaluator_error_is_rc1`), letting Bash callers tell "broke" from "graded" — a signal OMC's in-process loop does not expose. No adoption needed; recorded as a parity-or-better half-mechanism the prior pass elided.

**Circuit breakers / plateau detection / hard max-iteration backstop — ABSENT, verified CONFIRMED.**
Prior said: the deadline was omx's only autonomy lever; iteration/plateau bounds were not examined separately. Actually: omx's loop has no max-iteration backstop and no plateau detection — the only stop is the wall-clock deadline (`loop.py:39`) plus the SKILL's "no `--max-runtime` → run exactly ONE iteration" default (`SKILL.md:59`). That single-pass default is a real safety net, but a `--max-runtime` loop with a fast evaluator can churn hundreds of discard iterations against an un-improving checkpoint. OMC carries a dual cap (`hardMaxIterations=500`/200-strict, self-extending soft +10, `MAX_TODO_CONTINUATION_ATTEMPTS=5`) at `persistent-mode/index.ts:1163-1184`. Adoption: `loop-status` already reads the ledger — add a `plateau` field computing consecutive-discard count and no-improvement-since-N; `SKILL.md` step-7 stops when consecutive discards exceed a small N. Pure read over the existing ledger, no new state.

### state-paths

**5-stage state-root anchor resolution — ABSENT, verified CONFIRMED (minor evidence nit).**
OMC resolves its state root through a 5-stage ladder (env dir > `.omc-workspace` marker > git superproject climb > git toplevel > cwd) with a `$HOME`-stop guard and `OMC_DISABLE_MULTIREPO` kill switch (`15-lib-config-state.md:7-24`). omx has no root resolution at all: `OmxPaths.__init__` stores the caller-supplied `--root` verbatim (`omx_paths.py:169-173`). In this repo's own 3-layer install stack and worktree experiments a user must hand-supply the right anchor every call, and two worktrees of one repo get two disjoint `.omx/` trees with no shared identity. (Evidence nit from verification: the audit's "`--root required` on every subcommand" overstates — 13 of ~19 subparsers take `--root`; `ingest`/`reduce`/`session-id`/`eval` take `--path`. The substantive claim — no root resolution, verbatim storage — holds.) Adoption: `omx_core/root.py::resolve_omx_root(explicit=None)` implementing a Python-idiom ladder (explicit → `$OMX_STATE_DIR/<project_id>` where `project_id` = `{dirname}-{sha256(git-remote-or-path)[:16]}` → `.omx-workspace` marker climb, stop before `$HOME` → `git rev-parse --show-toplevel` → cwd), making `--root` optional. Keep OMC's placement/identity split (`15:197`): the identity hash never climbs submodules. Pure Python, no hook.

**Unbounded TensorBoard ingest — ABSENT (bounded reads), verified CONFIRMED — the most significant new correctness finding.**
OMC caps ingested state (`DEFAULT_PAYLOAD_LIMITS maxPayloadBytes=1MB`; `session_search` streams line-by-line; 5MB replay cap) at `15-lib-config-state.md:117` and `04-tools-state-memory.md:95,99`. omx ingests TB event files with `size_guidance={SCALARS:0}` (`tensorboard.py:30-32`) — TensorBoard's explicit sentinel for "keep every scalar, no reservoir downsampling" — then `Reload()`s the whole file into numpy with no cap, no timeout, no row limit, in BOTH the adapter and the skill's copy-paste guidance (`exp-analyze/SKILL.md:81`); `eval_summary.py:19` also `read_text()`s the whole JSON. A corrupt or multi-GB event file OOMs or hangs the analysis session with no loud failure. This directly contradicts the repo's own `.claude/rules/03` "Multi-run Trajectory Comparison" rule mandating stride-based extraction (verified present). The asymmetry is telling: `run_evaluator` already re-implements OMC's execution-timeout idea by passing a timeout to `subprocess.run` (`evaluator.py:69,85`), proving the bounding pattern is in-scope — it was just never applied to the ingest path. Adoption: change `size_guidance` to a reservoir cap (e.g. `{SCALARS:10000}`) or add `--max-scalars`/`--max-bytes` flags with an `st_size` pre-check that loud-fails past a `MAX_INGEST_BYTES` constant; wire into `omx ingest`/`plot`/`reduce`; update the `SKILL.md:81` snippet. Pure adapter change, tested in `test_ingest_tensorboard.py`.

**Scratch/state reaping + tombstones — ABSENT, verified CONFIRMED.**
OMC reaps state (cancel-signal-before-delete, completed-session orphan sweep, 24h SQLite cleanup, `MAX_REPLAY_FILES=10`) at `04-tools-state-memory.md:47-48` and `15-lib-config-state.md:116,181`. omx has no cleanup lifecycle: `scratch/<session_id>/` is created per session (`omx_paths.py:206`) and never garbage-collected — every `exp-analyze` session leaves a permanent scratch tree. There is no `omx clean` verb (verified: no `rmtree`/`trash`/`tombstone`/`reap` in `cli.py`). Adoption: `omx clean --scope session --session-id <sid>` (trash after a dry-run diff, honoring the repo's trash-not-rm rule) and `omx clean --stale --older-than 7d`, reusing the existing dry-run→approve pattern from wiki gc. Pure filesystem verb, no hook.

**Durability-atomic writes — PARTIAL, verified CONFIRMED.**
OMC's atomic-write recipe opens the temp `wx` (O_CREAT|O_EXCL) mode 0o600, `fsync`s the fd, renames, then best-effort `fsync`s the parent dir (`15-lib-config-state.md:113`). omx's `atomic_path`/`atomic_dir` (`omx_paths.py:373-424`) are rename-atomic via `os.replace` with careful exception cleanup, but there is zero `fsync` anywhere in omx-core, no `O_EXCL` on the temp (plain `with open`), and a fixed `.tmp` name so two writers to one target race on the temp itself. A power loss in the write window can lose a just-committed write. Adoption: after `yield tmp` and before `os.replace`, `os.fsync` the temp fd; after replace, best-effort `os.fsync` the parent dir; suffix the temp with `os.getpid()`+uuid. ~8 lines, tested in `test_omx_paths.py`. Low urgency for a workstation harness, as noted.

**project-memory / PreCompact-recovery analog — NA_QUESTIONABLE, verified CONFIRMED.**
OMC re-injects project-memory after compaction so directives survive a mid-task context reset (`04-tools-state-memory.md:75`). omx cannot hook PreCompact, but the underlying need — a long `exp-analyze` losing accumulated findings on compaction — is real; the partial mitigation (findings land in `report.md` incrementally, wiki is queryable) exists but there is no single resume pointer the skill re-reads. Adoption (no hook): a `scratch/<session_id>/notes.md` convention (path already exists, `omx_paths.py:215`) that `exp-analyze` appends `[FINDING]` tags to, plus a skill step "on resume, first cat scratch notes.md and any partial report.md" — the omx-idiom equivalent of PreCompact re-injection.

### knowledge

**Capture-then-curate — prior said ABSENT; verified CORRECTED to PARTIAL.**
Prior said: capture-then-curate is ABSENT. Actually: omx HAS it, but split by role-within-one-pass rather than by time. The core deterministically extracts `[FINDING]` candidates read-only (`cli.py:515-529`); the LLM then chooses durable ones and writes them (`exp-analyze/SKILL.md:407-435`, honoring `--no-wiki`). The write path and the judgment path ARE separated per the same design principle as OMC's time-split (SessionEnd dumb stub → next-session LLM promotes). But the audit's "HAS" overstates parity, and the verifier corrected it to PARTIAL: the OMC mechanism as titled is an unconditional, budget-bounded, LLM-free SessionEnd stub that fires regardless of what the session did. omx has no session-agnostic dumb-capture path — persistence requires `exp-analyze` to run AND the LLM to voluntarily write. A raw session that never invoked `exp-analyze` evaporates. That residual is exactly the prior gap's open point. Adoption: `omx wiki capture-session --from-report <r> --confidence low --category session-log` that DOES write (append-merge) every parsed finding as a low-confidence stub — the OMC SessionEnd analog — so a session that skipped the curate step still leaves breadcrumbs that lint's low-confidence + near-dup checks surface for promotion. Distinct from the curated `omx wiki add`, which stays LLM-judged.

**`--from-report` is a read-only extractor, not an auto-ingest — PARTIAL, verified CONFIRMED.**
Prior said (#4): `omx wiki add --from-report --auto` would auto-ingest, since "the extractor already exists." Actually: `cli.py:517-529` shows `--from-report` calls `parse_findings` (`report.py:42`), PRINTS `{candidates:[...]}`, returns 0, and WRITES NOTHING. The write is a deliberate separate `omx wiki add` the LLM issues after choosing durable findings — intentional (keeps human/LLM judgment in the loop, no silent noise), not a missing `--auto`. The correct omx-idiom improvement is therefore the `capture-session` fallback verb above, not an `--auto` flag on `--from-report`.

**Numeric quality-gate scoring before persisting a learning — ABSENT, verified CONFIRMED — the single most transferable OMC-11 mechanism omx lacks.**
OMC scores a candidate learning's concreteness mechanically before persisting (start 100; −30 short problem, −30 short solution, −20 no triggers, −10 oversize, −5/−10 weak triggers; pass requires `score >= MIN_QUALITY_SCORE 50`) at `11-knowledge-lifecycle.md:93,115` / `validator.ts:13-67`. omx's ingest validates only category/confidence/title-nonempty (`ingest.py:42-47`); confidence is a free LLM-chosen label never computed from content. The `exp-analyze` SKILL prescribes "write conclusion AND evidence, never a bare label" (`SKILL.md:437-453`) but that is prose-only — a label-only, evidence-free page ingests silently at whatever confidence the LLM claims. Adoption: `omx_core/wiki/quality.py::score_page(content, tags) -> int` (penalize content < ~120 chars, no numeric token, no `file:field`/backticked-command source marker, generic tags), wired into `ingest_knowledge` as a NON-blocking signal — if `score < 50`, force confidence to `low` (so lint's low-confidence check flags it) and log the score. Keeps report-only philosophy (never rejects); surface `QUALITY_FLOOR=50` as a named constant. Makes the "evidence not label" rule mechanical instead of prose.

**Skillify / recipe-promotion — NA_QUESTIONABLE, verified CONFIRMED (overturns the prior NA).**
Prior said: skillify is dismissed under "no subagents/orchestration." Actually: skillify is knowledge-lifecycle promotion, not orchestration, and omx's telos ("the more you use it, the more it specializes") is exactly what it serves. omx accumulates `debugging`-category wiki pages (reusable diagnostic procedures, `SKILL.md:461-462`) but has no path to promote a recurring procedure into a first-class reusable analysis recipe (verified: grep for skillify/recipe/promote in omx-core yields only `promote-plots`, an image mover). Adoption: `omx wiki promote-recipe --slug <debugging-page>` that reads a high-confidence, oft-queried `debugging` page (using `log.md` query frequency, `query.py:105`, as the OMC-analog usage signal) and emits `.omx/recipes/<name>.md` — a structured symptom→verb/plot recipe `exp-analyze`/`exp-design` read as a checklist — gated on OMC's same 3 questions (not Googleable / workspace-specific / took real effort). CLI verb producing markdown, no hook/subagent.

**Auto-regenerated environment feed from a structured store — NA_QUESTIONABLE, verified CONFIRMED.**
OMC auto-projects `project-memory.json` INTO the wiki as a live `environment.md` page (regenerate when the store is newer, category:environment confidence:high) at `11-knowledge-lifecycle.md:60-63`. omx has a structured store OMC lacks — `.omx/profile/{metrics.yaml,evaluator.sh,launch.sh}` — but it never appears in wiki query results; the `exp-init` seed page is a one-shot manual analog that drifts silently if the profile changes (`exp-init/SKILL.md:172-183`). omx's reserved-file set is `{index.md, log.md}` only (`types.py:25`) — no `environment.md`. Adoption: `omx wiki sync-profile`, a reserved `profile.md` writer (like OMC's `writeEnvironmentUnsafe`) that regenerates from the profile when its mtime is newer, excluded from `write_page`/`list_pages` like index/log, called from `exp-init` after `omx init` and from `exp-analyze`'s wiki-query step. Keeps no-hooks: a CLI verb the skill calls.

**Structural-contradiction lint polarity+grouping mismatch — PARTIAL, verified CORRECTED.**
omx's contradiction lint (`lint.py:49-83`) matches OMC on the tag-across-categories rule but differs on the confidence rule two ways, not one. Polarity: omx's a-1 flags tag groups where EVERY page is high-confidence; OMC (`lint.ts:159-161`) flags coexisting high+low (a low page shadowing an authoritative one). Grouping axis (the correction): OMC's confidence sub-check groups by a 2-segment SLUG PREFIX (`lint.ts:150`), not by shared tag; omx keys BOTH sub-checks on shared tag (`lint.py:56-59`). So the omx adoption is an omx-idiom adaptation, not a literal port: add an a-3 rule flagging a shared-tag group whose pages span both high and low confidence, catching the "low page shadows the high conclusion" smell OMC catches (at slug-prefix scope). Pure addition, candidate-not-verdict, no new state — invariant-safe.

### intake-design

**Sealed-evaluator / no-self-grading guard — ABSENT, verified CONFIRMED — the single most severe miss in the whole audit.**
self-improve's `validate.sh` diffs the worktree against the improve-branch merge-base and hard-fails (exit 1) if any `sealed_files` entry (the benchmark) was modified, so the optimizer physically cannot regrade its own homework (`10-selfimprove-goals.md:82-88`). omx treats `evaluator.sh` as a plain profile file (`profile.py:191-205`, written via `shutil.copyfile`) that the loop invokes verbatim (`bash .omx/profile/evaluator.sh`, `exp-loop/SKILL.md:81`) with zero integrity check — grep of omx-core for `sha256`/`sealed`/`integrity`/`hashlib` returns nothing. The loop's own author can edit `evaluator.sh` mid-loop and undetectably move the score truth-source; `run_evaluator` (`evaluator.py:69`) just shells whatever command it is handed. This is exactly the self-modification attack `validate.sh` was built to stop, and omx is fully exposed — the whole loop trusts a score from a file nothing seals. Adoption: `omx eval --seal-check` records a `content_sha256` of `evaluator.sh` into the profile manifest at approval time (when `pending_approval` flips to false), and before each `omx eval` recomputes it and loud-fails (`OmxError('evaluator.sh modified since approval; re-approve as an explicit change')`) on mismatch; `exp-loop` step 3 calls it before grading. A CLI verb, no hook — makes the seal mechanical, not prose.

**Confidence-gated setup handoff — ABSENT, verified CORRECTED.**
Confirms the prior gap row and goes deeper: OMC's guard is a validated code contract with tests (`setup-contract.ts:3` `AUTORESEARCH_SETUP_CONFIDENCE_THRESHOLD=0.8` + two throw guards ~92/96, tested at `setup-contract.test.ts:25-44`), whereas omx's `pending_approval` is a bare boolean (`profile.py:73`) with no `evaluator_source`/`evaluator_confidence` field at all (`validate_metrics_schema`, `profile.py:27-77`). A low-confidence inferred evaluator a human rubber-stamps silently becomes the loop's score truth-source. Correction (applied per verdict): the prior gap file's claim — repeated verbatim in the audit's evidence — that "setup-contract.ts comment even cites omx parity" is FALSE; grep of the entire 124-line file for `omx`/`parity`/`sibling` returns zero, the file carries no such comment. The substantive verdict (ABSENT, boolean-only) stands. Adoption: extend metrics.yaml with `evaluator_source:'user'|'inferred'` + `evaluator_confidence:float`, loud-fail when `source=='inferred'` and `confidence<0.8` and `pending_approval` was cleared. Rides the exp-init Criteria dimension at near-zero cost. (Roadmap #10 folds in the placeholder-regex gate half.)

**Ambiguity scoring math is HAS-at-parity, not a gap — HAS, verified CONFIRMED (corrects a possible over-flag).**
The 3-dim weighted math (Goal 0.40 / Constraint 0.30 / Success-criteria 0.30, `ambiguity = 1 − Σ(score·weight)`, threshold 0.2, round caps 10/20) is implemented as prose the model executes (`exp-init/SKILL.md:38-50`) — but that is exactly OMC's own design (`08-planning.md:57` "executed by the model"; only the threshold is resolved in TypeScript). So omx is at PARITY here, not a gap. The one OMC element omx omits is code-resolved threshold substitution from settings (omx hardcodes 0.2 in the skill). Minor; optional `omx interview-threshold` verb resolving `ambiguity_threshold` from the profile would restore OMC's auditable "threshold: X (source: Y)" line without prompt drift.

**Structured mandatory failure-analysis — NA_QUESTIONABLE, verified CONFIRMED.**
self-improve makes a non-success result schema-invalid without a categorized `failure_analysis` object (enum oom/timeout/regression/logic_error/scope_error/infrastructure/benchmark_parse_error/sealed_file_violation, `10-selfimprove-goals.md:88`), forcing post-mortems that feed the next round's planners. omx records THAT an eval failed (`status=error` + a fixed-template `parse_error` string, `evaluator.py:97-108`) but not a categorized post-mortem, and `decision.py` turns any error into discard with a fixed-template note only (`decision.py:70-71`) — so a recurring failure mode (e.g. the repo's own eval-path trap) isn't structurally learned. Adoption: add a `failure_category` enum (timeout/nonzero_exit/no_stdout/parse_error) to `run_evaluator`'s error records and have `exp-loop` auto-append a low-confidence wiki `debugging` page tagged with it on an evaluator-error discard, feeding exp-design's Step 1 wiki query — the capture-then-curate pattern applied to failures.

**Durable cross-run campaign ledger — PARTIAL, verified CONFIRMED.**
omx HAS the append-only ledger shape at the single-run level (the 3-artifact ledger, `ledger.py`), but lacks ultragoal's CROSS-experiment durability: a campaign-level plan+ledger tracking a multi-experiment initiative (design→launch→analyze→next) across sessions (`10-selfimprove-goals.md:105-119`, `goals.json` + append-only `ledger.jsonl` per plan). grep for `campaign` across omx-core returns zero; a campaign today lives only in scattered `experiments/` dirs + the wiki, with no replayable event log of which experiments were tried/kept/abandoned. ## Deliberately not adopting

Carried forward from the prior gap file, adjusted for the overturned NA calls (the agents-catalog, HUD/notifications, and knowledge-promotion NAs are no longer blanket — their statusline/subagent/routing cores stay out, but the sub-mechanisms above are now roadmap items).

- **Custom MCP server / bridge** (`03-mcp-bridge.md`) — violates the zero-runtime-dependency and no-MCP invariants head-on; design §11 proves the persistent-kernel payoff is ~0 for "1 run = 1 load".
- **Hooks engine (Stop/PreToolUse/SessionEnd/PreCompact)** (`02-hooks.md`) — omx is intentionally hookless; every roadmap item is delivered as a CLI verb the skill calls, keeping the surface at "Bash runs Python". The PreCompact-recovery need is met by a voluntary scratch breadcrumb (#24), not a hook.
- **Autonomous orchestration modes** (autopilot/ralph/team/ultrawork, `06`/`07`) — out of scope (D6); the `exp-loop` deadline is the only autonomy and never auto-launches training (D4/B8).
- **Reviewer/critic as a subagent** (`12`) — the author≠reviewer separation IS adopted (#4), but as a fresh-context CLI-verb gate, never a spawned agent.
- **Model-routing / delegation-enforcement gate** (`12`, `19`) — no subagents to route or Task calls to inject a model into.
- **Vector/embedding search** (any) — hard constraint (design §9 build #8): keyword+tag+CJK only, matching OMC's own no-embeddings stance.
- **HUD statusline / notification transport** (`13`) — statusline and chat-injection stay out; only the on-disk completion-marker idiom is adopted (#7), no daemon or push transport.
- **Plugin install/update systems** (`01`) — install is claudebase's job (D7); only the version-sync fan-out and a read-only `doctor` self-check are adopted (#6, #19), not an installer.
- **Full-unattended training auto-launch** — explicitly forbidden with no override path in v0.1 (D4/B8); the "training start/stop is the user's" rule is load-bearing.
- **Committed-build CI guard** (`18-delta-engineering.md`) — NA now (pure interpreted Python, no bundle), flagged to adopt OMC's commit-but-PR-guard pattern IF omx-core ever ships a vendored/compiled artifact (design §307).
