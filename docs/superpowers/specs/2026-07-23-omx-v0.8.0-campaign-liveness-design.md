# omx v0.8.0 — Campaign Liveness (design)

## Problem (field evidence, 2026-07-23, /workspace/constrained-albc)

After ~6 weeks of real use the project could not answer "what is done and what is left":

- `campaign-list`: `baseline` (1 event), `p7_tail` (18), `teacher_baseline_posttam`
  (**6 planned proposals, 0 ledger events**).
- The 4 groups actually holding runs on disk (`teacher_baseline_opt`,
  `teacher_baseline_buoyfix`, `seed_floor_dgx`, `e3_dgxscale_buoyfix`) have **no campaign
  entry at all**.
- Meanwhile the wiki (335 pages) and the experiments tree (36 gated reports) thrived.

Controlled comparison inside one harness: the state stores that survived are exactly the
ones a hook surfaces every turn (`<omx-open-backlog>`) or a frequently-run verb writes as
a byproduct (`report-coverage` → produced-reports.jsonl). Campaign logging has neither —
verified: `grep -rl campaign hooks/` = 0 files; campaign verbs are wired only into
`exp-loop` (never used by this project) and one unenforced "When done" bullet in
`exp-design` (SKILL.md:282-286, nothing gates it; `proposal-lint` never checks it).

## Root cause (investigation verdict)

**Deliberate scoping whose hardening plan died.** Campaign was scoped to
exp-loop/exp-design by the R2 spec (docs/superpowers/specs/2026-07-06-…-r2-tree-governance
-design.md:192-199). Warn→hard promotion was deferred R3→R4→R5→R6 waiting for dogfood
data (CHANGELOG.md:435-437, 357-359, 302-304); the justifying sweep (2026-07-11, "zero
.omx roots on any machine") predates the field project's actual adoption (~2026-07-14),
and R6 never shipped — 0.7.0 pivoted to topic releases and fixed the *structurally
identical* wiki gap (needs-apply-before-retrain now REFUSES queue-launch,
CHANGELOG.md:135-177) without revisiting the campaign sibling. The changed assumption:
campaign liveness was designed to ride exp-loop; real projects run exp-analyze /
exp-design / queue-launch manually and never enter exp-loop.

Not the cause: user discipline (wiki/tree health falsifies it). Contributing, out of omx
scope: the workspace convention routing active plans to `.sp/plans/` (61+ untracked docs,
two competing human id schemes Z*/A*/B*/C*) — addressed separately by
`2026-07-23-plan-consolidation-prompt.md`; noted in Non-goals.

## Fix — three legs, byproduct-first

### 1. Byproduct writes (prevention; the main lever)

- **`report-coverage` (36 uses in the field)**: when the gate passes and stamps
  (cli.py:694-718), also record a campaign event — `record_analyzed(paths, report_path)`:
  derive `run_id` = `report.parents[2]`, `group` = `report.parents[3]` (shape already
  guaranteed by `is_analysis_report`), **auto-init** the group campaign if absent
  (plan marked `auto_initialized: true`), append event `analyzed` with
  `data={"report": path}`, **dedup by report path** (field evidence: the same report was
  gate-stamped 3× in one day). Advisory: failure warns on stderr, never fails the verb —
  same contract as the produced-reports append.
- **`queue-launch`**: after `queue_pending_launch` succeeds (cli.py:959-965), append a
  `launched` event via `record_launched(paths, proposal_id, run_id)` — appended to the
  campaign **whose plan.json planned that proposal_id** (search `.omx/campaigns/*/plan.json`),
  not to the run's group. This makes the join work across group transitions (the exact
  multi-group failure observed: proposals planned in `teacher_baseline_posttam`, runs landing
  in `teacher_baseline_buoyfix`). Dedup by (proposal_id, run_id). Unplanned proposal →
  no event + one stderr hint naming `campaign-plan-add`. Advisory, never fails the verb.
- New event type `analyzed` added to `EVENTS`. `campaign_status` already treats unknown
  events as non-terminal (campaign.py:126-130), so derived_status semantics are unchanged.

### 2. Drift detection (the check that would have fired in July)

- Core `campaign_drift(paths, schema, base)` in campaign.py: walk index-tree runs at
  declared depth (reuse `omx_core.tree.walk_runs`/`runs_at_declared_depth`), group =
  run parent dir name. Flag (a) `unregistered` — group has runs, no campaign dir;
  (b) `empty_ledger` — campaign dir exists, group has runs, ledger has 0 events.
  Report-only, `{"ok", "unregistered", "empty_ledger"}` (tree-audit output style).
- CLI verb `campaign-drift` (`--root`, `--adopt`). `--adopt` = one-shot remediation:
  init missing campaigns (marked auto_initialized) and append one
  `note {kind: "adopted"}` event to silence empty ledgers. Idempotent.
- **Conditional hook surfacing** (the piece that removes discipline from the loop):
  `_fetch_campaign_drift` in hooks/handlers.py, appended to `_assemble_route_context`
  after the backlog block. In-process lazy import of omx_core (fail-open when absent —
  poison-import test contract preserved), silent (`""`) when no drift / no root / no
  tree.yaml / any exception — the oms `scholar_resume_emit` zero-tax pattern. Resolves
  its root via the same `_resolve_backlog_root` helper (a second, independent call —
  a few ms of ladder cost, accepted to keep `_fetch_open_backlog`'s pinned signature
  untouched). Block is ≤6 lines naming the drifted groups (capped) and the exact fix
  commands.

### 3. Legibility (the id gap that bred Z1/A3/C2)

- `campaign-plan-add --label` — optional short human label stored per planned entry;
  `campaign-status` plan view includes it. The status output thus IS the
  label ↔ proposal_id mapping table.
- `campaign-init --predecessor <id>` — stored in plan.json, surfaced by
  `campaign-list`/`campaign-status`. Link only; no join logic across predecessors (YAGNI
  until a real consumer exists).

## Design constraints honored

- **Per-turn context**: drift block is conditional (empty in the healthy state), tiny when
  present; static route checkpoint unchanged (2KiB cap test untouched).
- **Byproduct over discipline**: both writers fire inside verbs the field project already
  runs daily; zero new required steps.
- **Backward compatibility**: existing 3 campaigns readable unchanged (new fields are
  optional; `analyzed` is additive; p7_tail's 18 events untouched). No schema migration.
- **No silent behavior change**: new events/inits are visible in verb stdout/stderr and
  marked `auto_initialized`/`source` in the data they write; CHANGELOG + SKILL.md updated.

## Non-goals (stated, not hidden)

- The `.sp/plans` workspace convention still competes for human plan prose; omx now keeps
  the *machine* plan/outcome state alive regardless. Convention change is the
  plan-consolidation job's scope.
- No hard gate forcing `campaign-plan-add` at exp-design time (R6's old warn→hard idea);
  byproduct + drift surfacing is the lighter lever — revisit only if drift recurs.
- `doctor` / `tree-audit` are not extended; drift has its own verb + automatic hook
  surfacing. (A doctor one-liner would be a third surface for the same fact.)
- Hook changes reach live sessions only after plugin cache refresh + restart (cache is a
  plain copy of the repo; hand-patching it is forbidden).

scripts/sync_version.py; CHANGELOG (Added/Changed/Verification/Notes); omha card version
sync (out-of-repo edit, surfaced in the release report); `omx doctor` + `omx card-check`

`campaign-status` unchanged on the existing campaigns.
