## 2. R1 scope (v0.2.0) — twelve items

Enforcement axis (the hook skeleton is built once here; R3 reuses it):

1. Fail-open hook runner skeleton + kill switches
2. PreToolUse report-guard (blocks hand-Edit of gated deliverables)
3. Integrity stamp + `omx report-verify` (roadmap #14)
4. Reviewer lane: `omx report-review` verb + read-only reviewer agent + exp-analyze Review step (#4, absorbing #26's claim-evidence check)
5. Evaluator seal-check (#0 — pulled forward from the loop cluster on severity)

Rigor axis (the user's emphasized daily surfaces):

6. Wiki numeric quality gate (#3)
7. `omx wiki capture-session` fallback write verb (#11)
8. `omx wiki sync-profile` reserved page (#17)
9. Contradiction lint a-3 rule (#23)
10. exp-design precision: `omx proposal-lint` + `omx probe-novelty`
11. Bounded TB/JSON ingest (#2)
12. Version-sync + `omx doctor` (#6 + #19)

Deferred out of R1: `promote-recipe` (#15) → R3; campaign ledger (#28) → R2.

Build-order constraints the task breakdown must respect: item 1 before item 2 (hook runner before the guard handler); within item 3, stamp (report-coverage extension) → `report-verify` → `report-parse` integration; item 4's `report-review` verb before the reviewer agent (the agent runs the verb first); item 5's `profile-seal` verb before the `omx eval` seal-check; item 12's `doctor` before any skill step-0 preflight reference.

## 3. R1 component design

### 3.1 Hook runner skeleton

- `hooks/hooks.json` (registered from `.claude-plugin/plugin.json`) + `hooks/run_hook.py <handler-name>`.
- Runner contract, re-implementing OMC's `run.cjs` pattern in Python: read hook JSON from stdin; dispatch to a handler module; **any exception, timeout (internal 3s budget), or malformed input → exit 0** (fail-open). `OMX_SKIP_HOOKS` (comma-separated handler names) short-circuits named handlers; `OMX_DISABLE=1` short-circuits everything.
- **No pip dependency**: the runner injects `${CLAUDE_PLUGIN_ROOT}/omx-core` into `sys.path` to import `omx_core`; if that fails it degrades to stdlib-only behavior (for report-guard: allow). Hooks must work — or safely no-op — before `omx doctor` would even pass.
- Handlers are pure functions (stdin dict → decision dict) covered by Claude-free pytest with JSON fixtures.

### 3.2 PreToolUse report-guard

- Matcher: `Edit|Write` tool calls whose `file_path` matches the gated-deliverable patterns from `omx_paths` constants: `analysis/<analysis_id>/report.md`, `report.ko.md`, and `manifest.json` (stamp tampering is blocked too).
- Decision: deny, with the reason string instructing the RE-analysis path ("gated deliverables are written only by the exp-analyze atomic_path writer; re-enter the skill with the old report as BASE") and naming the escape hatch (`OMX_SKIP_HOOKS=report_guard`).
- Invariant that keeps the legitimate path working: exp-analyze writes reports via Bash → `atomic_path`, never via the Edit/Write tool, so the guard never fires on gate-passing writes. This closes the 0.1.14 incident at edit time; intentional friction on one-character fixes is accepted (that was the incident's exact shape).
- Fail-open: unparseable hook input → allow.

### 3.3 Integrity stamp + `omx report-verify` (#14)

- Stamper = verifier, one call (the audit's "who verifies the verifier" finding): `omx report-coverage`, on `ok:true`, immediately writes into the analysis dir's `manifest.json`: `integrity: {file_sha256: {"report.md": …, "report.ko.md": …}, gates_passed: [...], omx_version}`.
- Path contract: `manifest.json` is the **sibling** of the report file — every producer/consumer derives it as `dirname(report_path)/manifest.json` (matches the existing layout `analysis_dir/{report.md, report.ko.md, manifest.json}`, `omx_paths.py:305-318`). No verb gains run_id/analysis_id arguments for this; `report-parse` keeps its `--path <report.md>` signature and derives the manifest the same way.
- New verb `omx report-verify --path <report.md-or-analysis_dir>`: recompute hashes; rc2 on mismatch, missing stamp, or empty `gates_passed`.
- Consumer-boundary enforcement: `omx report-parse` (the verb exp-design/exp-loop use to read a report) calls verify internally and loud-fails on a tampered/unstamped report — downstream skills cannot skip it.
- RE-analysis re-stamps under a new `analysis_id`; old dirs stay valid.

### 3.4 Reviewer lane (#4, absorbing #26)

Two layers; author ≠ reviewer without violating anything (the reviewer agent is read-only):

- Mechanical: `omx report-review --path <report> [--baseline <prior>] [--record-to <analysis_dir>]` — deterministic critic checklist: wall-of-text heuristic (numbers-per-paragraph), every `[FINDING]` carries `[EVIDENCE]`+`[CONFIDENCE]`, **HIGH-confidence numeric claims cite a code-exec source, not a PNG** (#26 folded in), sibling-section shape consistency, depth vs baseline. Output `{verdict: approve|revise, findings: [{section, issue, severity}]}`; `--record-to` writes `review.json` into the analysis dir.
- Judgment: `agents/report-reviewer.md` — read-only (no Write/Edit), fresh context. Runs the verb first, then judges what code cannot (prose quality, whether evidence actually supports the claim). exp-analyze gains a Review step after the When-done gates: dispatch the reviewer agent; `revise` → apply through the RE-analysis path.
- R1 records the review; it does not yet make review presence mandatory for consumption (over-gating guard; candidate for a later release).

### 3.5 Evaluator seal-check (#0)

- `omx profile-seal --root <r>`: records sha256 of `.omx/profile/evaluator.sh` and `.omx/profile/launch.sh` into `.omx/profile/seal.json` at approval time.
- The check lives **inside `omx eval`** (not a skippable flag) as a **preflight before the command runs**. Contract: `omx eval` gains `--root` (today it takes only `--command/--cwd/--timeout/--keep-policy/--last-kept-score` and knows nothing about the profile). The check compares the current sha256 of `.omx/profile/{evaluator.sh, launch.sh}` against `seal.json` — it never inspects the `--command` string (the command may or may not call evaluator.sh; the seal guards the profile files themselves, which are what skills are instructed to execute).
- Outcomes: seal mismatch → rc2 "evaluator.sh modified since seal; run `omx profile-seal` to re-approve as an explicit change". Seal absent, or `--root` not passed (legacy callers) → warning only in 0.2.0 (backward compat; promotion to hard-fail is a later release). exp-loop/exp-analyze skill text updates to pass `--root`.

### 3.6 Wiki quality gate (#3)

- `omx_core/wiki/quality.py::score_page(content, tags) -> (score, reasons)`. Start 100; penalties: body <120 chars (-30), no numeric token (-20), no source marker — `file:line`, backticked command, or `[EVIDENCE` (-20), generic-only tags (-10), weak title (-10). `QUALITY_FLOOR = 50` (profile-overridable).
- Wired at the **call boundary, non-blocking**: the `wiki add` CLI handler (and `capture-session`) computes the score before calling `ingest_knowledge` and overrides the supplied confidence to `low` when below floor — `ingest_knowledge`'s contract (caller decides confidence, `ingest.py` docstring) is unchanged. `quality_score` + reasons recorded in frontmatter for lint to surface. Never rejects a page — INV-1 (lint reports, never auto-fixes) and INV-2 (append-merge, no loss) preserved.

### 3.7 `omx wiki capture-session` (#11)

- `omx wiki capture-session --from-report <report.md> [--session-id <sid>]`: writes every parsed `[FINDING]` as a low-confidence stub page (category `session-log`, tags `[auto-captured, <run_id>]`) via append-merge. This is the missing *write* half of the read-only `--from-report` extractor.
- Called unconditionally from exp-analyze's When-done and exp-loop step 7 — duplicate-vs-manual-curation is absorbed by slug merge. Guarantee: a session that skips curation still leaves breadcrumbs; lint's low-confidence check queues them for promotion.

### 3.8 `omx wiki sync-profile` (#17)

- Reserved page `profile.md` (reserved set becomes `{index.md, log.md, profile.md}`; excluded from `write_page`/`list_pages`). Regenerated from `.omx/profile/*` when the profile mtime is newer; category `environment`, confidence `high`.
- Called by exp-init after init and by exp-analyze's wiki-query step — the profile is always current in wiki queries, killing the one-shot seed-page drift.

### 3.9 Contradiction lint a-3 (#23)

- New rule in `lint.py`: a shared-tag group whose pages span both `high` and `low` confidence → candidate flag (report-only). omx-idiom adaptation of OMC's "low page shadows the authoritative conclusion" smell (OMC keys it on slug prefix; omx keys both existing sub-checks on tags, so a-3 stays on the tag axis).

### 3.10 exp-design precision verbs

- `omx proposal-lint --path <proposal.md>` (rc2, wired into exp-design When-done): (a) **discriminating-prediction contract** — the proposal must state the leading hypothesis H1 and the strongest alternative H2 with probe predictions that are both present and **different** (identical predictions = not a discriminating probe); (b) evidence tags on diagnosis claims; (c) references to the grounding analysis/ledger ids. The SKILL.md gains an "Alternatives table (H1/H2/discriminating prediction)" authoring rule — prose, but its presence is enforced by the lint.
- `omx probe-novelty --proposal <p>` (loud-warn, never fail — novelty is judgment): keyword-overlap search over wiki + past proposals + ledger decisions; reports "this probe family was tried in <id>, outcome discard".
- Together these mechanize the self-approval gap the audit found in exp-design (`SKILL.md:108-163` authors and approves its own probe).

### 3.11 Bounded ingest (#2)

- `tensorboard.py`: replace `size_guidance={SCALARS: 0}` with a reservoir cap constant (default 10_000; `--max-scalars` override) + `st_size` pre-check (`MAX_INGEST_BYTES` default 1 GiB; `--max-bytes` override); over-limit → rc2 with guidance. Same pre-check for `eval_summary.py`'s whole-file `read_text()`.
- Update the copy-paste snippet at `exp-analyze/SKILL.md:81`; fixture tests for both limits. Defaults are profile-overridable (D12).

### 3.12 Version-sync + `omx doctor` (#6 + #19)

- `scripts/sync_version.py`: SSOT = `plugin.json` version; fan-out targets = `pyproject.toml` (required) and `marketplace.json`/README version markers **only where a marker already exists** (never inject a new one — README currently has no version string). `test_version_sync.py` asserts plugin.json == pyproject.toml so drift fails pytest. Release checklist: run sync, then tests.
- `omx doctor [--root]`: read-only JSON `{omx_version, python_version, omx_core_importable, deps, profile_present, hooks_installed}`. Skills call it as a step-0 preflight so a stale/missing install fails actionably instead of as a raw shell error.

## 4. Cross-cutting

**Data flow (one exp-analyze cycle after R1).** bounded ingest → reduce/plot → report written via `atomic_path` (the only gate-passing write; Edit is hook-blocked) → `report-coverage` gates + stamps in the same call → `report-review` (mechanical) → reviewer agent (judgment) → revise loops through RE-analysis → `capture-session` + manual curation (quality-gated) → downstream `report-parse` verifies before parsing → proposal → `proposal-lint` + `probe-novelty` → launch queued for the human (D4).

**Error-handling philosophy.** CLI verbs: loud-fail rc2 (unchanged; stdout stays pure JSON, message on stderr). Hooks: fail-open exit 0, per-handler skip via `OMX_SKIP_HOOKS`, global `OMX_DISABLE`. The one blocking hook (report-guard) blocks a single tool call, never the session, and has the documented escape hatch.

**Testing.** All new logic Claude-free pytest on top of the existing 135: handler fixtures (stdin JSON → decision), stamp/verify and seal round-trips, quality score table cases, capture-session idempotency, proposal-lint true/false-positive fixtures, ingest limit fixtures, version-sync drift test.

**Distribution boundary (per-release acceptance gate, D12).** Mechanisms + default constants in core; thresholds overridable via profile; specialization only in `.omx/`. Check: fresh project + `omx init` = fully functional; no workspace-specific identifiers in core.

**Backward compatibility.** Unstamped legacy reports: verify passes with a "unstamped legacy" warning in 0.2.0 (rc2 promotion earmarked for 0.3.0). Absent seal: warning → later hard-fail (same pattern). No retroactive quality scoring of existing wiki pages. Hooks require plugin reinstall — the claudebase installer update (D7) is part of the R1 release checklist.

**Risks (accepted).** report-guard's friction on trivial edits is intentional (the 0.1.14 incident shape); reviewer-agent quality is LLM-dependent but floored by the mechanical layer; hooks.json schema is Claude-Code-version-coupled but fail-open + verb duplication means breakage degrades to v0.1 behavior, never blocks; PyPI-vs-vendored packaging stays open (R5).
