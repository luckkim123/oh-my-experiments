# omx v0.9.0 — Program Layer (design)

## Problem (field evidence, 2026-07-23, /workspace/constrained-albc)

The ALBC teacher campaign spans 4 run groups, each with its own group-keyed campaign
store. The cross-group program view — "what is the whole campaign's state" — lives in
`constrained-albc/docs/reference/teacher-campaign-plan.md` (520 lines) only because omx
defines no program-level artifact. That document's section 2 records why an umbrella
campaign cannot work: `campaign_status` derives `runs[]` from group-keyed ledger events,
so a campaign whose id matches no run group shows zero runs forever.

The linking key already exists and nothing consumes it: 3 of the 4 member campaigns
carry `program: "teacher-final-closeout"` plus `predecessor` pointers in plan.json
(written via `campaign-init --predecessor` / `extra` on 2026-07-23; the root campaign
`teacher_baseline_posttam` has neither — it IS the root). User direction 2026-07-23:
ALL experiment records, the plan included, belong under `.omx` — one root.

## Fix — a program artifact plus one read verb

### 1. Home: `.omx/programs/<program-id>/{PLAN.md, program.json}`

- Git tracking follows the store's existing policy: constrained-albc tracks `.omx`
  wholesale (208 files incl. campaigns + registry; only `.wiki-lock`/`scratch/`/`state/`
  ignored), so `.omx/programs/` is tracked with zero gitignore changes.
- `PLAN.md` = human/agent narrative, plain prose. Sessions append/edit in place. omx
  never parses, merges, or slug-manages it (unlike wiki pages). The ALBC document moves
  in VERBATIM at migration time.
- `program.json` = machine header, written only by `init_program` (atomic_path):
  `{"program_id", "campaigns": [...], "status": "active", "created"}`.
  Member list in program.json is the membership SSOT (fixes the root-campaign gap where
  plan.json's `program` field is absent). No `updated` field — nothing rewrites
  program.json after init (YAGNI until an edit verb exists).
- `program_id` validated with the same charset as campaign_id (`validate_run_id`),
  mirroring `OmxPaths.campaign_dir`.

### 2. Core (campaign.py — same domain as the store it reads)

- `init_program(paths, program_id, campaigns, *, now)`: refuse if the program dir
  exists; refuse if `campaigns` is empty or any member campaign dir is missing (fail
  loud — the field flow always has campaigns first); mkdir + write program.json; do NOT
  create PLAN.md (the migration `git mv` supplies it; a template stub would be
  overwritten anyway).
- `program_status(paths, program_id=None)`: resolve id (given id must exist; `None`
  resolves to the only program, else error listing available ids). Returns
  `{"program_id", "status", "plan_md": bool, "campaigns": [per-member campaign_status
  augmented with derived plan-entry counts]}` — one `campaign_status` call per member,
  no new derivation logic. A member campaign that disappeared since init surfaces as an
  `{"campaign_id", "error"}` entry instead of killing the whole view.
- `list_programs(paths)`: names of dirs under `.omx/programs/` containing program.json
  (used by resolution + CLI). Not a standalone verb — YAGNI.

### 3. CLI (cli.py)

- `omx program-init --id <id> --campaigns a,b,c [--root]` — creates the artifact,
  prints the JSON header. stdout reminds the caller that PLAN.md is supplied by the
  documented migration procedure (`git mv`), not generated.
- `omx program-status [--id <id>] [--root]` — JSON to stdout, same style as
  `campaign-status`. The ALBC section-0 contract becomes "PLAN.md + `omx program-status`".
- No `program-adopt` verb: the prompt allows a documented procedure and ponytail picks
  it — omx verbs do not shell out to git, and the migration is three commands
  (`program-init`, `git mv`, stub commit) documented in skills/docs.

## Design constraints honored

- **Auditor awareness (verified, not assumed)**: `tree_audit` walks the experiments
  tree only (zero `.omx` references), `doctor` checks only `.omx/profile/` files,
  wiki `gc` operates on the registry storage root. None flags `.omx/programs/` as
  foreign. A regression test pins this: with a populated `.omx/programs/`, tree-audit
  stays ok and wiki gc reports no candidates from it.
- **Campaign store schema unchanged**: program layer only READS campaigns
  (`campaign_status`); plan.json/ledger.jsonl untouched. p7_tail's 18 events and all
  v0.8.0 byproduct behavior unaffected.
- **No hook injection**: the optional per-turn program one-liner is SKIPPED — the
  backlog and campaign-drift blocks already exist, and program state has a pull verb.
  Revisit only if the field shows program state going stale despite the verb.
- **No silent behavior change**: purely additive verbs; existing verbs untouched.

## Non-goals (stated, not hidden)

- Results stay in the experiments tree; wiki unchanged; no multi-program dashboards,
  no HTML, no PLAN.md parsing/linting, no program-level ledger.
- No consumer for plan.json's `program`/`predecessor` fields beyond display —
  program.json membership is authoritative; cross-checking the two is deferred until a
  real mismatch occurs.
- The known field drift on `albc_trpo_student`/`teacher_baseline_opt` campaigns is
  pre-existing and out of scope.
