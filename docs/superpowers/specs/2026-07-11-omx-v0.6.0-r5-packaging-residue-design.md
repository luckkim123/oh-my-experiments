## 0. What R5 is and is not

R5 closes the **packaging/durability/hygiene residue** that every earlier round
explicitly deferred, plus the one loop-robustness residual that is NOT
dogfood-conditioned (the ledger-corruption circuit backstop, D-R4-4). R5 ships
**no new experiment semantics**: everything here is a test, a guard, a stub, a
clock, or a release-discipline verb.

**Dogfood precondition status (checked 2026-07-11, unchanged from R4):** the
real-data dogfood soak has still never run — zero `.omx` roots on any machine.
Everything D-R4-1 conditioned on soak data stays deferred (§6).

## 1. Decisions

### D-R5-1 — Skill-budget regression test (#18)

`omx-core/tests/test_skill_budget.py`: every `skills/*/SKILL.md` must be
**< 56 KiB (57,344 bytes)** and the corpus total **< 128 KiB (131,072 bytes)**.
Today: max = exp-analyze 44,602 B (~78% of ceiling), total = 87,163 B (66%).
The audit sketched 48 KiB, but that is 91% consumed on day one and R6
explicitly adds promotion/dogfood prose to exp-analyze — a guard that trips on
the next legitimate addition trains reflexive ceiling bumps (critic F5). 56 KiB
signals "real regression", not "normal growth". The test discovers skills by glob (no hardcoded list), so a
fifth skill is auto-covered. Failure message must name the offending file, its
size, the ceiling, and the documented remedy: *split reference prose into a
`references/` resource file read on demand* (the audit's sketch; the compaction
machinery itself stays NA — no installer to run it).

**Why:** `exp-analyze/SKILL.md` accreted gates through 0.1.7→0.5.0 and loads
whole into context on every invocation; nothing guards the growth. A pytest
ceiling is the contract-test-as-tier idiom omx already uses (#25).

### D-R5-2 — Deprecation-as-runtime-redirect for `omx wiki delete` (#20)

A real `wiki delete` subparser whose handler ALWAYS loud-fails (rc 2) with a
machine-readable redirect on stderr-path semantics identical to every other
loud-fail (SystemExit(str) → rc 2, stdout stays pure): the message is the JSON
`{"error": "deprecated", "reason": "wiki is append-merge (INV-2); removal is
git-guarded gc", "cli_replacement": "omx wiki gc --root <root> … then omx wiki
gc-apply"}` with the caller's actual `--root` echoed back (default `.` when
omitted). The stub accepts and ignores a positional `slug` and `--root`, so
every plausible mistaken invocation parses and reaches the redirect instead of
dying in argparse with a generic "invalid choice".

**Why:** 0.1.13 added the passive half (gc-apply help text) after a session
concluded "delete does not exist" from `--help` alone; OMC's stronger idiom
actively catches the deprecated call and reconstructs the replacement. ~10
lines, preserves the no-delete invariant (INV-2) — the stub deletes nothing.

### D-R5-3 — Durability fsync in the atomic-write helpers (#21)

`omx_paths.atomic_path` gains the durability half of write-atomicity: on the
success path, **before** `os.replace`, open the tmp file read-only and
`os.fsync` it; **after** the replace, open the parent directory and `os.fsync`
it (macOS/Linux both support directory fsync via `os.open(dir, os.O_RDONLY)`).
`atomic_dir` gets the parent-directory fsync only (recursively fsyncing every
file in a promoted analysis dir is not worth the IO; ceiling documented in the
docstring). `save_state`, the ledger writers, the loop marker, pending-launch,
and all wiki writes inherit the fix for free — they all route through
`atomic_path` (verified). The one raw `os.replace` outside the helpers
(`tree_ops.py:347-351`) is a **symlink** swap with a pid-suffixed tmp name — it
is NOT migratable to `atomic_path` (you cannot write a symlink into a yielded
file path, fsync-of-a-symlink follows to the target, and the pid suffix is a
real concurrent-aliaser guard that no coarse lock covers — critic C2). It stays
bespoke and untouched: a symlink swap has no file data to lose; the rename is
the durability unit. No fsync added there (YAGNI).

`atomic_dir`'s docstring must state honestly: the parent-dir fsync makes the
*rename* durable, NOT the promoted directory's file *contents* — a power loss
can still lose file data inside a just-promoted analysis dir (critic F7). That
is the accepted ceiling.

Note for the test contract: suite tests monkeypatch `os.fsync`, so the
843-baseline timing is not a durability-cost signal; the real cost (two fsyncs
per state write inside the Stop-hook lock) is accepted as single-digit ms on a
workstation.

The temp name stays `<name>.tmp` — NOT uuid-suffixed. Every `atomic_path`
caller that can race is already serialized by a coarser lock (state mutex, wiki
lock, run lease) — tree aliasing, the one unserialized racer, keeps its own
pid-suffixed bespoke swap (above). `atomic_dir` relies on the fixed `.tmp` name
for leftover cleanup. A uuid suffix would trade a non-existent race for orphan
accretion. (`# simplified:` comment at the tmp-name line naming this ceiling.)

**Why:** rename-atomic but not durability-atomic — a power loss in the write
window can lose a committed ledger row. Low urgency for a workstation harness,
but the fix is ~8 lines in ONE choke point.

### D-R5-4 — `omx card-check`: cross-repo card-currency guard

New verb `omx card-check [--card <path>] [--plugin-root <path>]`. Card path
resolution: `--card` flag → `OMX_CARD_PATH` env → documented default
`~/.claude/plugins/marketplaces/heroacademia/cards/omx.json`.

Plugin-manifest resolution is card-check's OWN (critic F3 — `doctor` never
reads plugin.json and its `plugin_root` has no fallback): `--plugin-root` flag
→ `CLAUDE_PLUGIN_ROOT` env → **repo-root fallback**
`Path(__file__).resolve().parents[N] / ".claude-plugin" / "plugin.json"` (the
`test_version_sync.py` idiom) — the editable install puts `__file__` inside the
repo, so release machines resolve without any env. Loud-fail rc 2 if no
plugin.json is found at the end of the ladder.

Checks:

1. `card["version"] == plugin.json["version"]` — the drift that is live today
   (card 0.1.0 vs plugin 0.5.0).
2. Every skill in `plugin.json["skills"]` — which are PATH strings like
   `"./skills/exp-init/"`, so **basename-strip first** (critic F3) — appears as
   a substring somewhere in the card JSON text. The audit sketched
   `card.triggers.skills == plugin.json skills`, but the real card's `skills`
   array is a *routing-lane* list, not a plugin-skill list — an equality check
   against the actual schema would always fail. Substring presence is the
   honest, schema-agnostic contract (verified: the live card's `triggers.skills`
   carries the exact bare names, so a correct card passes).

rc 0 + `{"ok": true, "card_version": …}` on parity; rc 2 with a per-check
`failures` list on drift; rc 2 with an actionable "card not found at <path>;
pass --card or set OMX_CARD_PATH" when the file is absent (a release machine
without omha installed must fail loudly, not vacuously pass — the verb exists
to be run at release time, where the card must be reachable).

Wiring: a "run `omx card-check`" line in the CHANGELOG release-discipline
header (the same place the version-bump rule lives) — NOT a pytest (the card
lives outside this repo; CI on a clean clone can't see it, and a skip-if-absent
test is a vacuous test).

**Why:** the live card says 0.1.0 with a pre-wiki description across 5
releases; `test_version_sync.py` guards the in-repo fan-out but nothing watches
the cross-repo copy. Note: card-check *detects*; updating the card is an
omha-repo edit outside R5's release scope (surfaced in the release report).

### D-R5-5 — Clock unification: one aware-UTC helper (`_now_iso` hygiene)

New module `omx_core/clock.py`:

- `now_iso() -> str` — `datetime.now(timezone.utc).isoformat()` (AWARE UTC).
- `parse_iso_utc(value, label) -> datetime` — loud-fail ISO parse that
  **normalizes**: a naive value gets `tzinfo=UTC` attached. Safe because both
  legacy writers were UTC instants (cli `_now_iso` was aware-UTC with tzinfo
  stripped; evaluator was aware-UTC) — normalization is exact, not a guess.

Migration rule — **instants unify, naming stays local, THE WIKI STAYS NAIVE**
(three categories, critic C1):

- All *stored/compared instant* sites OUTSIDE the wiki move to
  `clock.now_iso()`: cli.py `_now_iso` (deleted; non-wiki callers rewired),
  evaluator.py `_now_iso` (deleted), every inline
  `datetime.now(timezone.utc).isoformat()` in cli.py, loop.py, handlers.py
  (non-wiki paths).
- **Wiki instants stay naive-UTC by contract** — `ingest.py:41-42` actively
  REJECTS aware timestamps (`"now must be a naive ISO timestamp"`), and
  `lint._parse_iso` strips tzinfo to subtract naively; this is the wiki's
  deliberate on-disk format, not an accident. `clock.py` therefore also exports
  `now_iso_naive()` (= aware UTC with tzinfo stripped — same instant, wiki
  format), and the wiki-feeding call sites keep/switch to it: `cli.py:1374`
  (wiki add), `:1407` (capture-session), `:1420` (capture-flush), `:1443`
  (wiki note), `:1461/:1507/:1519/:1551` (gc/sync/lint/gc-apply `now`),
  `:778` (query), and `hooks/handlers.py:111-112` (report_guard capture).
  `ingest.py:41-42` and `lint._parse_iso` are NOT touched — the naive contract
  and its enforcement stay.
- All *parse-and-compare* sites move to `clock.parse_iso_utc()`:
  `loop._parse_iso` (delegates or is replaced), lock-lease age math
  (`lock.py:57-79` — normalize BOTH `armed_at` and `now` so a pre-R5 naive
  on-disk lease takes the real subtraction path, not the mtime fallback), and
  any other `fromisoformat` on a stored NON-wiki instant (SDD task does the
  exhaustive `grep -rn "fromisoformat\|utcnow\|datetime.now"` survey; the plan
  lists the sites found).
- *Naming* timestamps (`strftime` ids: tree ts naming cli.py:445, scratch
  analysis id cli.py:1584) stay local-naive BY DESIGN — human-facing directory
  names, never compared; each gets a one-line comment saying so.
- `deadline_passed`'s naive/aware-mix loud-fail STAYS (defense in depth), but
  after unification both sides are aware by construction. The `loop-arm --now`
  aware-only validation STAYS (rejects a naive injection at the entry point).
- Legacy compat test: a pre-R5 naive `armed_at` lease must age correctly
  against the new aware `now` via the normalizer (real subtraction, not mtime
  fallback); a naive `queued_at` in an existing pending-launch.json must read
  cleanly. The wiki needs no compat test — its format does not change.
- Grep-guard allowlist (the "no stray clock" test must NOT flag the correct
  implementation): clock.py itself, the two strftime naming sites, the wiki
  naive contract sites (`ingest.py`, `lint.py`), and `lock.py`/`loop.py` where
  parsing is delegated to `parse_iso_utc`.

**Why:** R4's mtime-fallback clock-mix was a *critical* plan-verification
finding, and the two-`_now_iso` split (naive cli vs aware evaluator) plus
"AWARE UTC — never _now_iso()" comments scattered at every gate call site are
the standing root cause. One helper, one parse, zero per-site vigilance.

### D-R5-6 — Circuit backstop that survives ledger corruption (D-R4-4 residual)

The R4 gate's circuit branch reads `ledger.json` at Stop time; any read failure
skips the branch (fail-open), so **a corrupted ledger disables its own
backstop**. R5 makes the backstop survive corruption with two additive
mechanics, both inside the existing state-mutex critical section:

1. **Health mirror at record time.** `record_iteration` (the write verb that
   just produced a healthy ledger) computes `loop_health` over the entries it
   just wrote and, when a loop is armed for that `run_id`, stores
   `{"consecutive_discards", "consecutive_faults", "at_iteration"}` as
   `active_loop["health_mirror"]` (under the state lock, same
   load→mutate→save idiom the gate uses). Sequencing (critic F4 note): the
   mirror write runs AFTER the ledger `atomic_path` writes complete — the
   ledger writes are never nested inside the state lock. Reentrancy is safe:
   `record_iteration`'s sole caller is `_cmd_run_record`, which holds no state
   lock. The CLI `run-record` path is the only writer; when no loop is armed
   it writes nothing.
2. **Corrupt-probe counter in the gate.** The gate's circuit branch narrows
   its exception handling: *absent* ledger (normal before the first record)
   → skip silently as today; *corrupt* ledger (`OmxError` whose cause is a
   JSON parse failure — `read_run_ledger` distinguishes the two today by
   message; give them distinct exception subtypes or a flag so the gate does
   not string-match) → increment `active_loop["ledger_probe_failures"]`,
   **write `state["active_loop"] = env` and `save_state` BEFORE returning**
   (critic F4 — the gate runs inside the state-lock critical section, and
   without an explicit save the increment dies with the Stop event: the
   cross-session path returns at the adoption check and never reaches the
   iteration-advance save). At **3** consecutive corrupt probes disarm with
   the NEW reason `"ledger_corrupt"`. Any successful probe resets the counter
   to 0 — and that reset is likewise `save_state`d (a stale 2 must not survive
   a healthy probe). While the counter climbs, the gate first consults the
   mirror: if the mirrored streaks already trip a threshold, disarm with the
   usual `plateau`/`fault_circuit` reason (the mirror reflects the last healthy
   read — corruption cannot rewind it).

`mark_loop_done`'s documented reason set grows to eight
(`done|deadline|cancel|error|hard_cap|plateau|fault_circuit|ledger_corrupt`);
`loop-disarm --reason`'s existing `choices` list (cli.py:1908-1910, six values)
gains `"ledger_corrupt"` as the seventh. `loop-mark-done`'s parser stays
deliberately narrower (`done|deadline|cancel|error`) — `ledger_corrupt` is an
armed-gate-only reason, matching the existing narrowing comment (critic F6). Any other circuit-branch exception keeps today's
blanket fail-open — the fail-open invariant (D9) is untouched; R5 narrows one
failure class into a counted, bounded stop.

**Why:** the verb path (`loop-health`, rc 2) is authoritative but
skill-voluntary; the gate is the backstop, and the one input that should trip
it hardest — a ledger so broken it cannot be read — currently switches it off.
Stopping when blind matches omx's loud-fail philosophy (a wrong loop is worse
than a stalled one). Threshold 3 (not 1) because a single corrupt read can be a
transient torn read from a concurrent writer on a non-atomic filesystem view;
three consecutive Stop-cycles of corruption is structural.

### D-R5-7 — PyPI-vs-vendored: DECIDED — neither; editable install via claudebase stays

The design §307 open question closes: `omx-core` ships on PyPI **no**, vendored
wheel in the plugin **no**. The install path remains claudebase's
`git clone + pip install -e` (design D7). Rationale: a single-user research
harness with zero external consumers gains nothing from PyPI (credential
custody, dist hygiene, release friction on every R-train round) and a vendored
wheel resurrects the committed-build problem the audit flagged OMC for. The
decision is recorded here + a CHANGELOG line + the design doc §307 marked
answered (one-line edit, `docs/design/2026-05-30-omx-experiment-harness-design.md`).
Revisit trigger (documented, not scheduled): a second consumer or a
multi-machine install pain that claudebase cannot absorb.
