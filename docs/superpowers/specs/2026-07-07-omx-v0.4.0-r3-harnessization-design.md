## 1. R3 scope — eight items

1. `omx-routing` UserPromptSubmit hook — sibling-family STAGE checkpoint for experiment work (omd `route_emit.py` MVP pattern).
2. SessionEnd capture hook + `omx wiki capture-flush` verb + `produced-reports.jsonl` write-site.
3. Post-compaction pointer hook — `SessionStart` matcher `compact` (read-only durable-state injection; audit #24 adapted per D-R3-6).
4. Stop-hook thin loop gate + `omx loop-arm` / `omx loop-disarm` verbs + `loop-state` in the `state.py` envelope (de-vestigializes `active_loop`).
5. Agents: `proposal-reviewer`, `wiki-curator`, `campaign-auditor` + dispatch steps in the owning skills.
6. `omx wiki promote-recipe` (#15) + `.omx/recipes/` consumption steps in exp-analyze/exp-design.
7. T11 — `campaign-status` raw `FileNotFoundError` normalized to the rc-2 loud-fail idiom.
8. Release mechanics: plugin.json hook registrations, version 0.4.0 fan-out via `scripts/sync_version.py`, CHANGELOG, README, contract-test extensions.

## 2. Component design

### 2.1 `route_emit` handler (UserPromptSubmit)

A static, stdlib-only checkpoint string (Korean, matching the sibling family), injected as
`{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext": CHECKPOINT}}`.
Content contract (mirrors omd's MVP — no keyword parsing, no dynamic generation):

- Fires the judgment line only for experiment-analysis work:
  `STAGE(exp) → <exp-init|exp-analyze|exp-design|exp-loop|wiki|tree|recipe> · <one-line reason>` — printed right after the omha ROUTE line; the block is ignored entirely for non-experiment work.
- Names the stage vocabulary and one-line meanings (init=profile bootstrap / analyze=runs→report / design=next-experiment proposal / loop=semi-autonomous cycle / wiki=knowledge ops / tree=output-tree governance / recipe=procedure promotion).
- Restates the three non-negotiables at routing time: training launch is never auto-fired (D4, queue only); reports are consumed via `omx report-parse`, never hand-parsed; results SSOT is the experiments tree.
- Knowledge bodies stay in `skills/*/SKILL.md` — the checkpoint names stages, it never embeds procedure (no-drift rule, omd precedent).

Checkpoint size is capped by test at 2 KiB — routing context is paid on every prompt in every project with the plugin enabled.

### 2.2 SessionEnd capture: write-site, verb, handler

**Write-site.** `_cmd_report_coverage`'s gate-passing stamp path (the CLI branch that calls `integrity.stamp_report`, cli.py:615) appends one JSON line `{"report": <abs path>, "stamped_at": <now>}` — `<now>` reuses the `_now_iso()` value already in scope at the stamp site (cli.py:614), so ledger and manifest carry the same stamp — to the root-level ledger `.omx/state/produced-reports.jsonl` (new `omx_paths.produced_reports_ledger()` getter — deliberately NOT under `scratch/`, whose paths mandate a session id the verb does not have). Append-only (`open(..., "a")` single-line writes; a torn last line is tolerated by the reader, which skips unparseable lines with a stderr warning, consistent with wiki's corrupt-page tolerance).

**Verb.** `omx wiki capture-flush [--root ...]` — reads the ledger, deduplicates report paths, and for each report that still exists calls the existing `capture_session` core (`wiki/capture.py:19`) exactly as `omx wiki capture-session --from-report` would. Missing report files are skipped and counted (`skipped`), never fatal. After processing, the ledger is atomically truncated (every entry was either captured or permanently gone); the small race window against a concurrently-stamping session is accepted for a workstation harness and noted in the verb docstring. Output: rc 0 + `{"captured": n, "skipped": m}`. Unreadable ledger *file* (not merely a torn line) → stderr warning + rc 0 with zero counts: flush is a rescue path and must not loud-fail at session end. Idempotence: `capture_session` is append-merge on a title-derived slug, so re-flushing the same report is a no-op merge — truncation is an optimization, not a correctness requirement.

**Handler.** `capture_flush(payload)` — lazily imports omx_core (report_guard precedent), anchors root resolution at `payload["cwd"]`, calls the same core function the verb calls, returns `None` (no decision output). Returning `None` is not merely style: the platform ignores SessionEnd hook output entirely (side effects only), so the flush's whole effect is the file-side capture. If omx_core is missing → `None` (fail-open). Registered `SessionEnd` with `"async": true` and a per-handler budget override: `run_hook.py` gains a module-level `_BUDGETS = {"capture_flush": 20}` table consulted before `signal.alarm` (default stays 3 s).

Platform grounding for `"async": true` (verified against docs v2.1.202, "Run hooks in the background"): the field is documented for `type: "command"` hooks, with the constraint that async hooks cannot return decisions — irrelevant here, since SessionEnd output is ignored by the platform regardless. Live precedent: OMC registers both its SessionEnd hooks `"async": true` on this machine. **The release assumes async is honored.** Fallback if a Claude Code version ignores the flag: the flush runs synchronously and shutdown is delayed by at most the 20 s budget — accepted worst case, no design change needed.

### 2.3 Post-compaction pointer handler (`compact_breadcrumb`)

Platform grounding (verified against docs v2.1.202): PreCompact hooks can only *block* compaction — they carry no `additionalContext` channel. The documented injection point is `SessionStart` with `source: "compact"`, which fires right after a compaction and DOES support `additionalContext`. The handler is therefore registered `SessionStart` / matcher `compact` and keeps the name `compact_breadcrumb` (it exists because of compaction).

`compact_breadcrumb(payload)` — read-only per D-R3-6. Composes a short block from three cheap sources (each individually guarded; any failure omits that line):

1. Recent scratch notes: glob `scratch/*/notes.md` with mtime < 48 h → list paths.
2. Armed loop: the `state.py` envelope's `active_loop` (§2.4) if armed → run_id + deadline.
3. Queued launches: `runs/*/pending-launch.json` present → run ids awaiting approval.

Injected as SessionStart `additionalContext`:
`<omx-durable-state> this session was just compacted — re-read: <the lines above> — scratch notes are the analysis breadcrumb trail; pending launches require the human gate; an armed loop is resumed only per exp-loop SKILL. </omx-durable-state>`
Empty state (nothing armed/queued/no notes) → handler returns `None` (inject nothing — silence over noise).

### 2.4 Stop-hook thin loop gate + arm/disarm verbs

**State.** The vestigial `state.py` envelope (`active_loop: None`, load/save already implemented and tested) becomes the carrier:
`active_loop = {"run_id", "armed_at", "deadline", "iteration", "hard_cap", "adopted_session"}`.

**Verbs (guarantee layer, D9).**
- `omx loop-arm --run-id <r> --max-runtime <sec> [--hard-cap N]` — computes `deadline = compute_deadline(now, max_runtime)` (CLI injects the real clock, same as queue-launch), writes the envelope. `--max-runtime` is **mandatory**: an armed gate always self-expires — this is the staleness guard that lets the gate be workspace-scoped. Arming while already armed → rc 2 loud-fail (one loop per root until R4's lock; disarm first). Default `hard_cap` 50.
- `omx loop-disarm [--reason done|deadline|cancel]` — clears `active_loop`, prints the final `{iteration, reason}`. Idempotent (disarming an unarmed root is rc 0 `{was_armed: false}`).
- `omx loop-status` extends its JSON with the armed envelope (pure read, existing verb).

**Handler.** `loop_gate(payload)` on Stop:

```
env kill-switches (runner-level)            -> allow (return None)
state unreadable / active_loop is None      -> allow (fail-open)
deadline_passed(deadline, now)              -> disarm(reason=deadline), allow
iteration >= hard_cap                       -> disarm(reason=hard_cap), allow
adopted_session set and != payload session  -> allow (another session's loop)
else -> adopt session if unset; iteration+=1; save;
        block: {"decision": "block", "reason": CONTINUATION(iteration)}
```

- **Session adoption**: the first session blocked after arming writes its Claude `session_id` into `adopted_session`; all other sessions pass through untouched. This resolves the workspace-scoped-state vs. multi-session ambiguity deterministically without an identity mapping.
- **Ownership semantics (intended, not incidental)**: an armed loop OWNS the adopted session until disarm or expiry — *every* stop of that session is continued, including turns where the user asked something unrelated mid-arm. This is what persistent mode means (OMC ralph behaves identically); the alternative (a per-turn scope signal) would re-grow the gate toward the state machine D-R3-1 rejected. The exp-loop SKILL's arm step states this to the user explicitly ("arming dedicates this session to the loop until `omx loop-disarm`"), and the continuation prompt's disarm sentence is the standing exit.
- **Crashed adopted session self-heals**: if the adopted session dies without disarming, nothing continues the loop, and the armed state simply expires at the mandatory deadline (next stop in any session after expiry triggers the disarm-and-allow branch). No reaping logic is needed in R3.
- **Clock discipline**: `loop-arm` and the handler both use the *aware* UTC clock (`datetime.now(timezone.utc).isoformat()`, matching `_cmd_loop_status`), never the naive `_now_iso()` used by wiki/stamp paths — `deadline_passed` loud-fails on a naive/aware mix (loop.py:50), and inside the handler that exception fail-opens the gate, silently killing the loop. A handler test covers the mixed-clock case.
- **Continuation prompt** (fixed template, iteration substituted): continue the exp-loop cycle per `skills/exp-loop/SKILL.md` — analyze → design → eval → decide → log; queue any launch via `omx queue-launch` (**never execute a launch — D4**); when the deadline passes or the work is done, run `omx loop-disarm`. The D4 sentence is part of the frozen template and covered by test.
- `stop_hook_active` is deliberately *not* an allow-condition — repeated blocks across turns are the loop. Runaway is bounded by the two hard conditions above (deadline mandatory at arm time; hard_cap default 50).
- **exp-loop SKILL.md** gains step 0 "arm" (`omx loop-arm` with the user-approved max-runtime) and extends step 7 "stop" (`omx loop-disarm --reason done` — deadline expiry needs no action, the gate self-disarms). The skill body's cycle logic is unchanged (D-R3-1).

### 2.5 Agent catalog (three additions)

All three mirror `agents/report-reviewer.md`: read-only frontmatter (`tools: Read, Grep, Glob, Bash`, `model: sonnet`), Bash restricted to the named omx verbs + read-only inspection, mechanical-layer-first procedure, JSON-only final message, and the calling session applies changes through the owning skill's path (never the agent).

| Agent | Mechanical layer (runs first) | Judgment layer (what code cannot check) | Output |
|:---|:---|:---|:---|
| `proposal-reviewer` | `omx proposal-lint --path <proposal>` + `omx probe-novelty` | Does the probe actually *discriminate* — do the leading hypothesis and the strongest alternative predict different outcomes? Is it truly one-variable? Are numeric findings carried with provenance? | `{"verdict": approve\|revise, "mechanical", "judgment": [...], "summary"}` |
| `wiki-curator` | `omx wiki gc` (read-only diagnosis) + `omx wiki lint` | Which flagged pages are truly dead vs. dormant; which near-duplicates merge in which direction; drafts the `## DELETE`/`## MERGE` proposal body with per-slug rationale | Proposal BODY as text — the session writes the `kind: wiki-gc` file, the human approves, `gc-apply` executes (existing two-phase path unchanged) |
| `campaign-auditor` | `omx campaign-status --id <c>` + `omx campaign-list` | Ledger hygiene: decisions recorded for every eval, probes novel vs. prior events, runs missing analysis reports, group naming consistent with conventions | `{"verdict": clean\|issues, "mechanical", "findings": [...], "summary"}` |

Dispatch wiring (skill edits, each a single inserted step):
- `exp-design/SKILL.md`: after the proposal artifact is written, dispatch `proposal-reviewer`; `revise` verdicts route through re-proposal by the authoring session (author≠reviewer, mirroring exp-analyze's report-reviewer step). This closes the audit's "exp-design self-authors and self-approves" finding (:111) — `proposal-lint` was the mechanical half, this is the judgment half.
- `exp-analyze/SKILL.md` wiki-maintenance section: gc runs point to `wiki-curator` for proposal drafting (replacing "hand-write the proposal").
- `exp-loop/SKILL.md`: optional `campaign-auditor` dispatch at the loop's stop step when a campaign id is set.

### 2.6 `omx wiki promote-recipe` (#15)

`omx wiki promote-recipe --slug <debugging-page> [--name <recipe-name>] [--force] [--root ...]`

- Loud-fails (rc 2): page absent; page category is not `debugging`; target `.omx/recipes/<name>.md` exists without `--force`.
- Mechanics: reads the page, computes the usage signal `query_count` with ONE defined parse — the number of `## [...] query` blocks in `registry/log.md` whose `**Pages:**` list contains `<slug>` (the format storage.py:226 writes; "returned this page", not "slug mentioned anywhere") — and writes `.omx/recipes/<name>.md` (new `omx_paths.recipes_dir()` getter) with frontmatter `{source_slug, promoted_at, query_count}` and the page body. Output: rc 0 + `{"recipe": <path>, "query_count": n}`.
- The OMC 3-question gate (not Googleable / workspace-specific / took real effort) is **prompt-side with a human gate**: the skills instruct asking the user before running the verb; the verb itself stays a mechanical, reversible file creation. Recipes are NOT gated deliverables (report_guard does not cover them) — the promoting session may restructure the recipe body (symptom → checks → plots → decision) after the verb runs.
- Consumption: `exp-analyze` and `exp-design` SKILL.md each gain one step — before diagnosis, list `.omx/recipes/` and follow any symptom-matching recipe as a checklist. No new list verb (a directory listing suffices — YAGNI).

### 2.7 T11 — campaign-status loud-fail hygiene

The unguarded read is the `plan.json` load at campaign.py:88 (`json.loads(paths.campaign_plan(campaign_id).read_text())`) — the ledger read at campaign.py:106 is already `.is_file()`-guarded. A vanished `plan.json` between init and status surfaces a raw `FileNotFoundError` (unreachable in normal flows — R2 final-review triage). Normalize to the standard `SystemExit(str)` → rc 2 + stderr message idiom, with a covering test that removes `plan.json` (not the ledger) between init and status.

### 2.8 plugin.json registrations

```
"hooks": {
  "PreToolUse":      [Edit|Write -> run_hook.py report_guard]        (existing)
  "UserPromptSubmit":[* -> run_hook.py route_emit]
  "SessionEnd":      [* -> run_hook.py capture_flush, "async": true]
  "SessionStart":    [compact -> run_hook.py compact_breadcrumb]
  "Stop":            [* -> run_hook.py loop_gate]
}
```

All five entries share the one runner; `OMX_DISABLE=1` silences all, `OMX_SKIP_HOOKS=<name,...>` silences each.

## 3. Cross-cutting

- **Testing (Claude-free, pytest)**: handler unit tests feed payload dicts and assert decision/None (per-handler: route_emit shape + size cap; capture_flush idempotence + corrupt-jsonl tolerance + missing-core fail-open; compact_breadcrumb empty-state silence; loop_gate full decision cascade incl. adoption, expiry-disarm, D4 sentence present in the frozen template). Verb tests mirror existing CLI test idioms (rc 0 JSON / rc 2 stderr). Contract tests: (a) new — plugin.json hook registrations ↔ `HANDLERS` ∪ built-ins parity, both directions; (b) existing #25 skill-to-CLI verb test automatically covers `loop-arm`/`loop-disarm`/`capture-flush`/`promote-recipe` once skills reference them; (c) new — `hooks/handlers.py` must import with omx_core absent (stdlib fallback survives, sys.modules patched).
- **Docs**: CHANGELOG 0.4.0 entry (incident-rooted style); README hooks section (five registrations, kill switches, the D4 sentence); `docs/HANDOFF.md` NEXT pointer update.
- **Version fan-out**: bump plugin.json → run `scripts/sync_version.py` (R1 #6) → pyproject/README markers; release task asserts `test_version_sync.py` green.
- **Execution prerequisite (before SDD dogfooding)**: reinstall the plugin + re-run `pip install -e` so the environment reflects v0.3.0 (R1/R2 quirk — hook activation and `omx doctor` version lag until reinstall). R3's own hooks are then testable live at the end of the branch the same way.
- **Live hook-fire acceptance check (release task, mandatory)**: Claude-free pytest cannot prove the platform actually fires the five events or honors the output schemas. After the end-of-branch reinstall, verify live: route_emit's STAGE block appears on a fresh prompt; loop_gate blocks a stop while armed and allows after disarm; capture_flush leaves session-log stubs after a session end; compact_breadcrumb's pointer appears on the first prompt after a compaction (SessionStart source=compact); SessionEnd shutdown is not visibly delayed (async honored). Record the observed results in the release-task report — this is the acceptance evidence for the platform-contract assumptions resolved at spec review (§2.2 async, §2.3 SessionStart-compact).
